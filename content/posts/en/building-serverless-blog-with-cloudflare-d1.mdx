# Building a Serverless Blog with Cloudflare D1 and Next.js

When it came time to build a new blog, I wanted something that was fast, global, and easy to maintain. After exploring various options, I settled on a combination that's been working beautifully: **Cloudflare D1** for the database and **Next.js** for the frontend.

## Why This Stack?

### The Problem with Traditional Setups

Most blog setups follow a familiar pattern:

1. A server (VPS or cloud instance)
2. A database (PostgreSQL, MySQL)
3. A backend framework
4. A frontend framework

This works, but it comes with overhead:

- **Server maintenance** - Security updates, monitoring, scaling
- **Database management** - Backups, connection pooling, migrations
- **Global latency** - Users far from your server wait longer

### The Edge-First Approach

Cloudflare's approach is different:

- **D1** - SQLite at the edge, no server management
- **Workers/Pages** - Your code runs close to users worldwide
- **Zero cold starts** - Unlike some serverless platforms

## Setting Up D1

D1 is Cloudflare's distributed SQLite database. Here's how I set it up:

### Creating the Database

```bash
# Using Wrangler CLI
wrangler d1 create blog
```

This gives you a database ID that you can reference in your configuration.

### Defining the Schema

```sql
CREATE TABLE posts (
  id TEXT PRIMARY KEY,
  title TEXT NOT NULL,
  slug TEXT UNIQUE NOT NULL,
  excerpt TEXT,
  content TEXT,
  date TEXT,
  likes INTEGER DEFAULT 0,
  created_at TEXT
);

CREATE TABLE reactions (
  id TEXT PRIMARY KEY,
  post_id TEXT NOT NULL,
  reaction_type TEXT CHECK(reaction_type IN ('like', 'dislike')),
  ip_hash TEXT NOT NULL,
  created_at TEXT,
  FOREIGN KEY (post_id) REFERENCES posts(id) ON DELETE CASCADE,
  UNIQUE(post_id, ip_hash)
);
```

### Querying from Next.js

With D1 bindings configured in Cloudflare Pages, querying is straightforward:

```typescript
// In your API route or server component
export async function GET(request: Request, env: Env) {
  const result = await env.DB.prepare(
    "SELECT * FROM posts ORDER BY date DESC"
  ).all();
  
  return Response.json(result.results);
}
```

## The Architecture

Here's how the pieces fit together:

```
┌─────────────────┐     ┌─────────────────┐
│   Cloudflare    │     │   Cloudflare    │
│     Pages       │────▶│       D1        │
│  (Next.js App)  │     │   (SQLite)      │
└─────────────────┘     └─────────────────┘
        │
        ▼
   Edge Network
   (300+ locations)
```

1. **Next.js** handles routing, SSR, and the UI
2. **D1** stores posts, reactions, and metadata
3. **Cloudflare's edge network** serves everything globally

## Performance Benefits

The results have been impressive:

| Metric | Before | After |
|--------|--------|-------|
| Time to First Byte | 200-500ms | 20-50ms |
| Database Query Time | 50-100ms | 5-15ms |
| Global Availability | Single region | 300+ locations |

### Why It's Fast

1. **Edge Execution** - Your code runs at the nearest Cloudflare location
2. **Local Database** - D1 replicas are distributed globally
3. **No Connection Overhead** - SQLite is embedded, no network roundtrips

## Multilingual Support

One requirement was supporting multiple languages (English, Chinese, Japanese, Korean). Next.js made this straightforward with `next-intl`:

```typescript
// src/i18n/request.ts
export const locales = ['en', 'zh', 'ja', 'ko'] as const;
export const defaultLocale = 'en' as const;

// Content structure
content/posts/
  ├── en/
  │   └── my-post.mdx
  ├── zh/
  │   └── my-post.mdx
  ├── ja/
  │   └── my-post.mdx
  └── ko/
      └── my-post.mdx
```

Push to GitHub, Cloudflare handles the rest. No CI/CD pipelines to maintain.

### 3. Instant Rollbacks

Every deployment is versioned. One click to rollback if something goes wrong.

## Trade-offs to Consider

No solution is perfect. Here's what to keep in mind:

### D1 Limitations

- **SQLite constraints** - No complex joins, limited concurrency
- **Database size** - Currently limited to 500MB per database
- **No real-time** - Not suitable for chat or live updates

### When to Use Something Else

Consider PostgreSQL or a managed database if you need:

- Complex relational queries
- Real-time subscriptions
- Large data volumes (>500MB)

## Conclusion

For a blog, documentation site, or content-focused application, the combination of Cloudflare D1 and Next.js is hard to beat:

- **Simplicity** - No servers to manage
- **Performance** - Edge execution and local databases
- **Cost** - Generous free tier, pay only for what you use
- **Developer Experience** - Familiar tools, instant deployments

The future of web development is edge-first, and this stack is a great way to get there.
